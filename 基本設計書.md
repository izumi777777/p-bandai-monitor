# PB Stock Monitor Pro (Auction Scout AI) 基本設計書

## 1. システム概要

### 1.1 目的
本システムは、ヤフオクの落札相場・現在開催中のオークション状況を自動で監視・鑑定し、ユーザーの仕入れ判断を支援する Web アプリケーションである。  
プレミアムバンダイの特定商品の在庫復活監視機能も補助的に提供するが、今後はヤフオク関連機能を主軸として進化させる方針とする。
手動でのリサーチ作業を AI とスクレイピング技術によって自動化し、ユーザーへ「最適な仕入れ・購入タイミング」を LINE でリアルタイムに通知することを目的とする。

### 1.2 想定ユーザー
- **コレクター・一般ユーザー**: プレミアムバンダイの限定商品、再販商品の在庫復活をいち早く知りたい層。
- **せどり・古物商ユーザー**: ヤフオクの過去相場と現在価格を比較し、利益の出る商品を効率的に仕入れたい層。

## 2. 全体アーキテクチャ

本システムは、以下の技術スタックを用いて構成されるSPA(Single Page Application)である。

- **フロントエンド**: HTML, Tailwind CSS, Vanilla JS
- **バックエンド**: Python (Flask)
- **データベース**: Firebase Firestore
- **認証基盤**: Firebase Authentication
- **AIエンジン**: Azure AI Projects / Agents (商品評価・提案・相場鑑定)
- **スクレイピング**: curl_cffi (TLSフィンガープリント偽装), BeautifulSoup4
- **ジョブスケジューラ**: APScheduler (バックグラウンドでの定期監視実行)
- **通知連携**: LINE Messaging API

## 3. 機能要件 (機能設計)

### 3.1 プレミアムバンダイ在庫監視機能（補助的機能）
- **高速スクレイピング**: 対象 URL からタイトル、価格、在庫ステータス（在庫マーク）、商品画像を取得する。
- **AI在庫コメント（プレビュー用途）**: スクレイピング結果を Azure AI Agent に渡し、在庫復活の有無や注意点を JSON 形式で解釈して返却する。現時点では Firestore への保存は行わず、画面上の「プレビュー表示」のみに利用する（将来的には機能縮小・廃止も検討）。

### 3.2 ヤフオク相場・AI鑑定機能 (Auction Scout)
- **検索キーワード最適化**: ユーザー入力をヤフオクの「あいまい検索」に最適化(全角/半角の正規化、スペース自動挿入)。
- **落札相場取得**: 過去の落札データを収集し、最高値・平均値・サンプル数を算出。
- **AIせどり鑑定**: 市場データをAzure AIに渡し、「推奨仕入上限価格」「収益性(Rank A/B/C)」「検品時の注意点」をJSON形式で出力。
- **AI追加チャット**: 鑑定結果に対し、ユーザーが文脈を引き継いだまま追加質問(例:「箱なしの場合は?」)を行える対話機能。

### 3.3 検索キーワード・サジェスト機能(入力補助／将来拡張・未実装)
- **将来拡張候補**: ユーザーの入力に対し、外部サジェスト API を用いてリアルタイムに検索候補を提案する機能を将来的な拡張として想定している（現時点の実装では未対応）。
- **APIフォールバック案**: 第一候補として Yahoo! 検索サジェスト API、失敗時には Google サジェスト API へ切り替えることで可用性を担保する設計とする。

### 3.4 開催中オークション監視・追跡機能
- **開催中リスト取得**: 指定キーワードで現在出品中の商品リスト(現在価格、即決価格、入札件数、残り時間)を取得。
- **個別ページ監視とLINE通知ロジック**:
  1. **予算超過通知**: ユーザーが設定したターゲット価格(my_target_price)を超過した場合に通知。
  2. **終了間近通知**: オークションの残り時間が10分以下になったタイミングで通知。
  3. **終了オークション自動処理**: オークションが「終了」状態となった場合、落札価格やメモ情報をユーザー専用のアーカイブコレクションに保存した上で、監視リストから自動削除する。

### 3.5 監視リスト一括登録機能
- **CSVアップロード**: 一行目に url 列を持つCSVファイルを解析し、最大5件までの一括監視登録を実行。
- **AIおすすめ提案登録**: Azure AI Agentが提案する「今注目すべき商品URLリスト(JSON)」をそのまま一括登録。

## 4. データ設計 (Firestore)

以下のパス構造で、ユーザー単位のセキュアなデータ管理を行う。

- **ユーザー設定**: `artifacts/{APP_ID}/users/{uid}/settings/line`
  - 保存項目:  
    - `lineUserId` (LINE Push 通知用)

- **監視リスト**: `artifacts/{APP_ID}/users/{uid}/watchlist`
  - 保存項目（表示・監視用の基本情報）:  
    - `url`（対象商品の URL・プレバンまたはヤフオク）  
    - `title`（商品名）  
    - `price`（表示用価格文字列）  
    - `inStock`（プレバン用の在庫フラグ）  
    - `statusText`（在庫状態・現在価格と残り時間などの説明テキスト）  
    - `imageUrl`（商品画像 URL）  
    - `memo`（ユーザー任意のメモ／検品ポイントなど）  
    - `my_target_price`（ヤフオク用のターゲット価格）
  - 監視用メタデータ:  
    - `createdAt`, `lastChecked`, `updatedAt`
  - 重複通知防止フラグ・状態フラグ:  
    - `last_notified_price` (前回「予算超過通知」を行った時点の価格)  
    - `notified_10min` (終了10分前通知済みフラグ)  
    - `lastNotifiedStatus` (プレバン在庫変動の最終通知状態)

- **ヤフオク終了オークション・アーカイブ**: `artifacts/{APP_ID}/users/{uid}/yahoo_archive`
  - 監視リストから削除される直前の情報を保存し、後から履歴として参照できるようにする。  
  - 主な保存項目:  
    - 監視リスト由来: `title`, `url`（または `sourceUrl`）, `imageUrl`, `memo`, `my_target_price`, `createdAt` など  
    - 終了時に付与される情報:  
      - `finalPrice`（最終価格の整数値）  
      - `finalPriceText`（「12,345円」形式の文字列）  
      - `endedAt`（終了検知時刻・サーバタイムスタンプ）

## 5. 外部連携・定期ジョブ設計

### 5.1 LINE連携 (Webhook)
- **ユーザーID取得**: ユーザーが公式LINEアカウントにメッセージを送信した際、Webhook経由で user_id を自動返信する。
- **Push通知**: バックグラウンドジョブが在庫変動や価格更新を検知した際、対象ユーザーへ即座にPushメッセージを送信。

### 5.2 定期実行ジョブ (check_watchlist_job)
- APScheduler を用いて、アプリ内でバックグラウンドプロセスとして稼働。
- 5分間隔で全ユーザーの watchlist を走査し、プレバンの在庫状態やヤフオクの現在価格・残り時間を更新する。
- ヤフオクについては、ターゲット価格超過や終了10分前を検知して LINE 通知を行い、オークション終了時には `yahoo_archive` へのアーカイブと watchlist からの自動削除を行う。

## 6. 非機能要件・セキュリティ

### 6.1 認証・認可
- **API保護**: Firebase Web SDKで取得したIDトークンを Authorization: Bearer ヘッダーで送信し、バックエンドの @login_required デコレータで検証する。
- **開発モード**: 環境変数 SKIP_AUTH=true により、ローカル開発時の認証バイパスを可能とする。

### 6.2 強力なBot対策(スクレイピング安定化)
- 一般的な requests によるアクセスではヤフオク等のWAF(403 Forbidden)で遮断されるため、curl_cffi (impersonate="chrome120") を採用。ブラウザのTLSフィンガープリントを精巧に偽装し、安定したデータ収集を実現。

### 6.3 SSRF(サーバーサイド・リクエスト・フォージェリ)対策
- URL登録時およびバッチ処理時に、許可されたドメイン(p-bandai.jp, yahoo.co.jp, E2Eテスト用の /test-item)以外への外部通信を遮断するバリデーションを実施。

## 7. 制約事項・今後の課題

- **スケジューラの多重起動**: 現在の APScheduler はアプリケーションメモリ内で動作するため、クラウド環境(AWS App Runner, Render等)でマルチワーカー構成にする場合、通知が重複するリスクがある。本番環境でスケールさせる際は、Cloud Scheduler等への外部切り出しを検討する。
- **APIレートリミット**: ヤフオクおよびAzure AIの呼び出し頻度が高まるにつれ、429(Too Many Requests)エラーの懸念がある。必要に応じてジョブ内に適度なディレイ(time.sleep)を設ける。
